# Provably Fair Pack Openings (B4)

This document describes the **intended** commit–reveal user flow and cryptographic transcript for pack openings. It satisfies the Part B requirement that users can verify outcomes were fixed fairly **at purchase time** (with pre-purchase commitment to server entropy) and that the platform cannot swap results after the fact without detection.

**Primitives:** standard SHA-256 and HMAC-SHA256 only (no custom crypto). Verification must be reproducible in the **browser** (Web Crypto).

---

## Terms

| Term | Meaning |
|------|---------|
| **`client_seed`** | String the user chooses (or accepts from the app). Influences the draw so the house cannot pick outcomes using only its own secret. |
| **`server_secret`** | High-entropy random string generated by the server. **Hidden** until the reveal phase. |
| **`server_commitment`** | `SHA256(server_secret)` (or an equivalent commitment). Published **before** reveal so the server cannot substitute a different secret later. |
| **`nonce`** | Opaque unique id (e.g. UUID) for a single “fairness session.” It links the pre-purchase commit to checkout and the eventual `user_packs` row. **Not** the English word “none.” |
| **Transcript / finalize-at-purchase** | The exact byte layout used inside HMAC (see implementation spec when built): must include ids that exist **only after** a successful purchase, e.g. `user_packs.id` and/or `pack_inventory.id`, plus `nonce`, `client_seed`, algorithm version, and a **`pool_fingerprint`** (hash of the ordered eligible pool) so catalog changes do not silently change verification. |

---

## Phase 1 — Commit (drop page, before the user owns a pack)

**When:** User is on the drop sale page (before or during the drop; before fulfillment creates their `user_packs` row).

**User flow**

1. User may enter a **`client_seed`** (optional). If they skip it, the server generates one so verification still has user-influence entropy without manual entry.
2. Client calls **`POST /drops/:dropId/fairness-commit`** (authenticated).
   - Body: `{ "client_seed": "..." }` — optional. Omitted, `null`, empty string, or whitespace-only means **server-generated** `client_seed`.

**Server behavior** (implemented)

1. Generate **`server_secret`** (32 random bytes; stored as hex in `pack_fairness_commit.server_secret_hex`).
2. Set **`nonce`** = new row primary key (`pack_fairness_commit.id`, UUID).
3. Compute **`server_commitment`** = `SHA256(server_secret)` (lowercase hex, 64 chars) → `server_commitment_hex`.
4. Resolve **`client_seed`**: use trimmed user string if non-empty (max 512 chars), else generate 32 random bytes as hex (64 chars). Set **`client_seed_source`** to `user` or `server`.
5. Persist **`pack_fairness_commit`**: `user_id`, `drop_id`, `client_seed`, `client_seed_source`, `server_secret_hex`, `server_commitment_hex`, `consumed_at` null, `user_pack_id` null (Phase 2).

**Response to client (do not return raw `server_secret`)**

- **`nonce`** — UUID; show in UI as “Fairness session ID.”
- **`server_commitment`** — 64-char hex; proves server entropy was fixed before reveal.
- **`client_seed`** — always returned (user-supplied or server-generated) so the client can store it for verification.
- **`client_seed_source`** — `"user"` | `"server"`.

**User-visible story:** *I locked in my seed (or accepted a random one); the server published a fingerprint of its secret; I have a session id to take to checkout.*

## Phase 2 — Purchase (finalize the cryptographic transcript)

**When:** Purchase succeeds: wallet debited, `pack_inventory` updated, **`user_packs`** inserted.

**User flow**

1. User completes the existing reserve / queue / pay path.
2. Client sends **`nonce`** (and drop context as needed) so the server can load the commit row for this user and drop.

**Server behavior**

1. Load **`fairness_commit`** by `nonce` + `user_id` + `drop_id`; reject if missing, expired, or already consumed.
2. After **`user_packs.id`** (and/or **`pack_inventory.id`**) is known inside the **same atomic transaction** as card fulfillment:
   - Build the **final** HMAC message per documented spec, including at minimum **`user_packs.id`** and/or **`pack_inventory.id`**, **`nonce`**, **`client_seed`**, slot indices, **`algorithm_version`**, **`pool_fingerprint`**.
   - Derive uniform indices (document modulo vs rejection sampling to avoid bias).
   - Write **`user_cards`** from the derived catalog card ids (this replaces or augments reading static `pack_card` for provably fair SKUs).
3. Mark the commit row **`consumed_at`**, link **`user_pack_id`**, enforce **one nonce → one successful purchase**.

**User-visible story:** *My earlier commitment was combined with the id of this specific pack purchase, so the result is not reusable for another pack.*

**Integration note:** Today, `packPurchaseQueueConsumer` resolves cards from `pack_card` on the pack template. A B4 implementation should either branch on “provably fair drop” or migrate to outcome derivation as described here for those drops.

---

## Phase 3 — Reveal (show verification inputs)

**When:** Pack opened or “Fairness details” viewed on a completed purchase.

**Server returns** (for the given `user_pack_id`):

- `client_seed`
- `server_secret` (**now** revealed — user checks `SHA256(server_secret) === server_commitment` from Phase 1)
- `nonce`
- `server_commitment` (or recomputed client-side from `server_secret`)
- `user_pack_id`, `algorithm_version`, `pool_fingerprint`, ordered outcome catalog card ids / external ids as needed for the verifier

**User-visible story:** *The server shows its real secret; I confirm it matches the fingerprint I saw before I bought.*

---

## Phase 4 — Verification page (browser-only)

**Requirement:** Verification logic runs **in the browser**; it must not rely on a server endpoint that returns only “valid: true.”

**Flow**

1. User pastes (or loads) commitment, `client_seed`, revealed `server_secret`, `nonce`, `user_pack_id`, algorithm version, pool definition or fingerprint.
2. Page uses **Web Crypto** (`crypto.subtle`) for SHA-256 and HMAC-SHA256, recomputes card ids.
3. User compares recomputed ids to the pack’s recorded contents.

Optional: append each opening’s non-secret fields to a **public audit log** for aggregate distribution checks (chi-squared / CIs vs advertised weights), aligned with B5.

---

## Why this ordering matters

| Step | Rationale |
|------|-----------|
| **Commitment before reveal** | The server cannot replace `server_secret` after outcomes are known without breaking `server_commitment`. |
| **Finalize with `user_pack_id` / inventory id** | Binds the draw to exactly one purchase; the same Phase 1 nonce cannot legitimately unlock two different packs. |
| **`nonce` as session id** | Stable handle from commit → purchase without ambiguous state; clear UX copy avoids “none” confusion. |
